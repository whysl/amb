## 1. 项目背景与核心原则

**客户**：威高商管公司（下辖民俗邨、时光城、威高广场三个运营主体）

**核心诉求**：各部门每月只需**填写一遍数据**，系统自动生成推移表、威高广场汇总表、商管公司汇总表，避免重复填写。

**核心原则**：
- 各部门数据相互独立，不存在数据引用关系
- 填报 → 部门审核 → 公司审核 三级流程，公司审核通过后才计入汇总
- 推移表 = 同一部门全年12个月数据横向展示
- 年度累计 = 1月至当前月实际数据之和（实时累加）

## 2. 组织架构

### 三级层级结构

```
Level 1: 威高商管公司（company）
├── Level 2: 民俗邨（minsuqun）
├── Level 2: 时光城（shiguangcheng）
└── Level 2: 威高广场（whgc）
    ├── Level 3: 招商中心（zhaoshang）
    ├── Level 3: 安全环境部·安全（safety）       ← 独立填报单元
    ├── Level 3: 安全环境部·环境（environment）   ← 独立填报单元
    ├── Level 3: 餐饮部（catering）
    ├── Level 3: 工程服务部（engineering）        ← 仅费用，无收入
    ├── Level 3: 营销推广部（marketing）
    ├── Level 3: 顾客服务部（service）
    ├── Level 3: 开乐影城（cinema）              ← 独立运营单元，结构特殊
    └── Level 3: 综合管理部（general）            ← 仅费用，无收入
```

**12个独立填报单元**：minsuqun、shiguangcheng、zhaoshang、safety、environment、catering、engineering、marketing、service、cinema、general（共11个 Level 3/2 部门；whgc 作为汇总层不直接填报）

> **注意**：安全环境部安全与环境是两个完全独立的填报单元，各自有独立填报人和审核人；财务部不参与填报，已从系统中移除。

## 3. 技术架构

| 层级 | 技术选型 | 说明 |
|------|---------|------|
| 后端框架 | Python FastAPI | 异步REST API |
| 数据库 | SQLite + SQLAlchemy ORM | 本地部署，单文件数据库 `amoeba.db` |
| 认证 | JWT（python-jose） | `sub` 字段强制转 `str`，避免 jose 兼容问题 |

## 5. 后端API清单

**统一前缀**：`/api`
**认证方式**：Header `Authorization: Bearer <JWT_TOKEN>`

### 认证模块 `/api/auth`
| 方法 | 路径 | 权限 | 说明 |
|------|------|------|------|
| POST | `/auth/login` | 公开 | 登录，返回access_token + user_info |
| GET | `/auth/users` | super_admin | 获取所有用户列表 |
| POST | `/auth/users` | super_admin | 创建用户 |
| PUT | `/auth/users/{user_id}` | super_admin | 修改用户 |

### 组织架构 `/api/org`
| 方法 | 路径 | 权限 | 说明 |
|------|------|------|------|
| GET | `/org/departments` | 已登录 | 获取所有部门列表（含层级、code、name） |
| GET | `/org/departments/{dept_id}/subjects` | 已登录 | 获取部门科目列表 |
| GET | `/org/my-department/subjects` | 已登录 | 获取当前用户所在部门的科目列表 |

### 填报表单 `/api/forms`
| 方法 | 路径 | 权限 | 说明 |
|------|------|------|------|
| GET | `/forms/my` | dept_filler+ | 获取我的填报列表（含period信息） |
| GET | `/forms/{form_id}` | 已登录 | 获取表单详情（含items数组） |
| POST | `/forms` | dept_filler+ | 创建/更新表单（自动保存草稿） |
| PUT | `/forms/{form_id}/submit` | dept_filler+ | 提交审核 |
| PUT | `/forms/{form_id}/dept-review` | dept_reviewer+ | 部门审核（approve/reject+comment） |
| PUT | `/forms/{form_id}/company-review` | company_reviewer+ | 公司审核 |
| GET | `/forms/pending/dept` | dept_reviewer+ | 待部门审核列表 |
| GET | `/forms/pending/company` | company_reviewer+ | 待公司审核列表 |

### 填报周期 `/api/periods`
| 方法 | 路径 | 权限 | 说明 |
|------|------|------|------|
| GET | `/periods` | 已登录 | 获取所有周期 |
| POST | `/periods/create-monthly` | super_admin | 创建月度周期（自动创建actual+planned+budget 3条） |

**创建月度周期参数**：`?year=2026&month=4`（month为系统月，自动推算上月actual）

### 汇总报表 `/api/summary`
| 方法 | 路径 | 权限 | 说明 |
|------|------|------|------|
| GET | `/summary/whgc` | company_reviewer+ | 威高广场当月汇总（需year+month+period_type参数） |
| GET | `/summary/company` | company_reviewer | 商管公司当月汇总 |
| GET | `/summary/rollup` | 已登录 | 推移表数据（dept_code+year+period_type） |
| GET | `/summary/rollup-compare` | 已登录 | 推移表对比（三类数据并排，dept_code+year） |
| GET | `/summary/dept-subjects` | 已登录 | 获取部门科目元数据（供推移表表头用） |

**rollup返回格式**：
```json
[
  {
    "month": 1,
    "subjects": {"A": 12345.0, "A1": 5000.0, ...},
    "cumulative": {"A": 12345.0, ...}
  },
  ...
]
```

**rollup-compare返回格式**：
```json
{
  "budget": [...],   // 同rollup格式
  "planned": [...],
  "actual": [...]
}
```

### Excel导出 `/api/export`
| 方法 | 路径 | 权限 | 说明 |
|------|------|------|------|
| GET | `/export/dept-report` | dept_filler+ | 导出单部门月度核算表（dept_code+year+month） |
| GET | `/export/rollup-report` | dept_filler+ | 导出推移表Excel（dept_code+year+period_type） |
| GET | `/export/company-report` | company_reviewer | 导出商管公司汇总报表（year+report_type+month/quarter） |

> **导出认证方式**：由于浏览器直接打开URL下载，token通过URL参数传递：`?token=xxxxx`（后端需从Query参数读取token，当前代码可能需要适配）

## 6. 前端页面清单

### 各页面功能说明

**Home.vue（首页）**
- 顶部欢迎Banner：显示用户姓名、所属部门、角色、今日日期
- 快捷功能Grid：根据角色动态显示（填报/推移表/审核/汇总/管理）

- 按科目category分组显示（收入/费用/时间/人工费等）
- is_calculated=true 的科目根据 formula 自动计算（前端实时计算）

- 显示当前审批状态，审核人可在此页驳回并写意见

**Rollup.vue（推移表）** ⭐核心功能
- 筛选：年份 + 类型（实际/预定/预算）+ 部门
- **单视图**：12个月横向滚动表格，科目按category分组，最后一列显示年度累计
- **对比视图**：预算/预定/实际三色并排（蓝/绿/红）
- 数值超过1万自动转换为万元显示（如 `12.34万`）
- 导出Excel按钮（调用 `/api/export/rollup-report`）

**Summary.vue（汇总报表）**
- 选择年月查询威高广场汇总 + 商管公司汇总
- 导出Excel

**Admin.vue（系统管理）**
- 创建月度填报周期（输入年份+月份）
- 用户管理列表
- 新建用户（用户名/密码/姓名/角色/部门）

## 7. 权限体系

### 四级角色

| 角色代码 | 名称 | 权限范围 |
|---------|------|---------|
| `dept_filler` | 部门填报人 | 只能填报自己部门的表单，查看自己的填报记录 |
| `dept_reviewer` | 部门审核人 | 在填报人权限基础上，可审核本部门表单（通过/驳回） |
| `company_reviewer` | 公司审核人 | 可查看所有部门表单，执行公司级审核，查看汇总报表 |
| `super_admin` | 超级管理员 | 全部权限，含用户管理、周期管理 |

### 数据隔离规则
- `dept_filler`/`dept_reviewer`：只能看到**自己部门**的表单
- `company_reviewer`：可看到**所有部门**的表单（待审核状态）
- 推移表：填报人只能查看**自己部门**的推移表（待实现）；公司审核人可查看所有

## 8. 各部门科目配置

> 科目已在 `init_data.py` 中完整配置，数据库初始化时自动创建。以下为各部门结构摘要。

### 民俗邨（minsuqun）
- 收入：A1回款、A2租金收入、A3物业费收入、A4兴源收入、A5研学团建、A6车场收入、A7能源收入 → A总销售额
- 变动费用：B1推广费、B2其他成本、B3招待费、B4差旅费 → B变动费用合计
- 边界利益：C = A - B
- 固定费用：D1-D7（能源/外包/维保/维修/办公/折旧/其他） → D固定费用合计
- 附加价值：E = C - D；G = E / F
- 时间：F1正常、F2加班、F3分摊 → F总时间
- 重要指标：J人数、K应收账款余额、L人均销售额
- 人工费：N人工费；O税前利润 = C - N；P税前利润率；Q单位时间人工费

### 时光城（shiguangcheng）
- 收入：A1-A6（租金/物业费/推广/停车/其他/能源） → A总销售额
- 费用：B1-B9 → B费用合计
- 附加价值：E = A - B；G = E / F
- 时间：F1/F2/F3 → F；重要指标：J回款额
- 人工费：N/O/P/Q

### 招商中心（zhaoshang）
- 收入：A1租赁收入、A2联营收入 → A
- 费用：B1-B5 → D费用合计（注意：合计字段用D而非B）
- 附加价值：E = A - D；G = E / F
- 时间：F1/F2 → F
- 重要指标：J月度签约额、L人均签约额、M威高广场出租率

### 安全环境部·安全（safety）
- 收入：A1停车（含子项A1a-A1e：线上/特来电/一点V来/积分/停简单）、A2月卡、A3其他 → A
- 费用：D1-D7 → D固定费用合计
- 附加价值：E = A - D

### 安全环境部·环境（environment）
- 收入：A1绿化租摆、A2景观灯电费、A3垃圾桶广告、A4场地租赁 → A
- 费用：D1-D5（绿化外包/水电/清洁工具/垃圾袋/清运外包） → D

### 餐饮部（catering）
- 收入：A1餐饮提成、A2其他 → A
- 费用：D1-D4（招待/差旅/办公/折旧） → D

### 工程服务部（engineering）⚠️ 仅费用
- 费用：D1-D19（电费公区/空调/供暖/水费/电费商户/维保/维修A馆B馆C馆/材料/办公/电话/团建/培训/折旧/招待/劳保/差旅/交通） → D固定费用合计
- 时间：F1/F2 → F；重要指标：J人数

### 营销推广部（marketing）
- 收入：A1威高广场客流（人次）、A2市场收益 → A月度销售累计
- 费用：D1-D12（活动/会员/新媒体/品牌/广告/差旅/培训/交通/通讯/办公/团建/折旧） → D
- 附加价值：E = A - D；时间：F1/F2 → F
- 人工费：F1_labor劳务费，N = F1_labor（汇总字段）

### 顾客服务部（service）
- 收入：A1 VIP卡、A2会员费、A3微信积分 → A
- 费用：D1-D3 → D

### 开乐影城（cinema）⚠️ 结构特殊
- 收入：A1-A4（票房/卖场/广告/其他） → A
- **变动费用**：B1-B5（票房成本/卖品/激光/电影专资/促销） → B；边界利益 C = A - B
- **固定费用**：D2-D5（能源/设备/其他/财务） → D（注意：没有D1）
- 附加价值：E = C - D（不是A-D！）

### 综合管理部（general）⚠️ 仅费用
- 费用：D1-D12（维保/福利/招待/差旅/通讯/办公/法务/系统建设/工会/团建/认证/培训） → D
- 时间：F1/F2 → F；重要指标：J人数

## 9. 填报与审核流程

```
[系统管理员] 创建月度周期
      ↓ 自动生成3条Period记录（actual + planned + budget）
[部门填报人] 进入 /forms，看到本月待填报表单
      ↓ 填写数据（支持草稿保存）
      ↓ 点击"提交审核" → status: draft → submitted
[部门审核人] 进入 /review/dept，看到待审核列表
      ↓ 通过 → status: dept_approved
      ↓ 驳回（附意见） → status: dept_rejected → 填报人修改后重新提交
[公司审核人] 进入 /review/company，看到已部门审核的列表
      ↓ 通过 → status: company_approved → ✅ 计入汇总数据
      ↓ 驳回 → status: company_rejected → 流程回退
```

**重要规则**：
- 只有 `company_approved` 状态的表单才会被计入推移表和汇总报表
- 同一部门同一周期只能有一份表单
- 填报人只能看到自己部门的表单

## 10. 推移表逻辑

**推移表**（Rollup Table）= 某部门全年12个月数据横向展示

### 数据结构
```
科目代码 | 科目名称 | 1月 | 2月 | ... | 12月 | 年度累计
```

### 计算规则
- **实际推移表**：取每月 `period_type=actual` + `status=company_approved` 的表单数据
- **预定推移表**：取每月 `period_type=planned` + `status=company_approved` 的表单数据
- **预算推移表**：取每月 `period_type=budget` + `status=company_approved` 的表单数据
- **年度累计**：= 1月至当前月的实际数据累加（只有actual有意义；预定/预算可按需求实现）
- 某月无数据（未填报/未审核）→ 显示 `None`，前端显示 `—`

### 汇总推移表（公司级）
- `dept_code=company`：商管公司全年各月汇总数字
- `dept_code=whgc`：威高广场全年各月汇总数字

### 对比视图
- 同时加载 budget + planned + actual 三组数据并排显示
- 颜色约定：蓝=预算、绿=预定、红=实际

## 11. 汇总计算逻辑

### 威高广场汇总（calc_whgc_summary）

汇总以下指标（取各下属部门 `company_approved` 表单的对应科目值求和）：

| 指标 | 来源部门和科目 |
|------|--------------|
| 总收入 | zhaoshang.A + safety.A + environment.A + catering.A + marketing.A + service.A + cinema.A |
| 总费用 | 以上各部门.D + engineering.D + general.D |
| 总附加价值 | 以上各部门.E（工程/综管无E） |
| 总时间 | 所有部门.F |
| 总人工费 | 所有部门.N |
| 单位时间附加价值 | 总附加价值 / 总时间 |

### 商管公司汇总（calc_company_summary）

```
总收入 = 民俗邨.A + 时光城.A + 威高广场总收入
总费用 = 民俗邨.D + 时光城.B + 威高广场总费用
总附加价值 = 总收入 - 总费用
威高广场单位时间附加价值（透传）
```

> ⚠️ 注意：时光城的费用合计字段是 `B`，不是 `D`

## 14. 已知问题与注意事项

### 已知Bug / 技术债

1. **Excel导出Token传递问题**
   - 当前 `window.open(url + '?token=xxx')` 方式导出，后端需要同时支持从 Query 参数读取 token（而不仅是 Header）
   - 当前代码中 `require_roles` 装饰器只读 Header，导出接口可能返回 401
   - **修复建议**：在 `auth.py` 的 `get_current_user` 依赖中，优先读 Header，如无则读 Query 参数 `token`

2. **公式计算前端实现不完整**
   - `FormDetail.vue` 中 `is_calculated` 字段应根据 `formula` 实时计算
   - 当前实现可能不完整，特别是 `E/F`（除法）类公式需注意除零保护

3. **推移表分类行显示逻辑**
   - `Rollup.vue` 的 `getLastCategory()` 函数用于判断是否需要显示分类分组行
   - 当切换部门或重新加载时，响应式更新可能有延迟，建议改为 computed 属性

4. **Admin.vue 部门选择器**
   - 已修复：van-picker 格式改为 `{ text, value }` 对象数组
   - 当前只显示 level=3 的部门，民俗邨(level=2)和时光城(level=2)未出现在新建用户的部门选择中
   - **待修复**：应显示所有可填报单元（level=2的民俗邨/时光城 + level=3的威高广场下属部门）

5. **威高广场（whgc）节点无科目**
   - `whgc` 是汇总层级，无独立科目，不参与填报
   - 访问 `/org/departments/whgc_id/subjects` 返回空数组，这是正常的

### 开发注意事项

- **部门编码大小写**：数据库中全部为小写英文（`whgc`、`zhaoshang`），前端不得使用大写形式
- **JWT sub字段**：创建Token时 `sub` 必须转为 `str`；解析时用 `int(payload.get("sub", 0))`
- **SQLite并发**：本地单用户场景，不需要处理并发；若多用户同时操作可能有锁等待
- **前端构建**：修改任何 Vue 文件后必须 `npm run build` 重新构建，否则页面不更新
- **跨域**：开发时 Vite 代理到 8000 端口；生产环境 FastAPI 直接服务 dist 目录，无跨域问题
